# 영수증 등록 사용자 가이드 (Phase 20 ERP Intake)

- **대상**: 컨설턴트 / 사장님 (일반 사용자)
- **버전**: Phase 20 MVP (2026-05-15)
- **관련 모듈**: Pack F. ERP → 영수증 등록 (`/erp/intake`) / 상품 (`/erp/products`) / 재고 (`/erp/inventory`) / 주문 (`/erp/orders`)
- **사양 원본**: [`./2026-05-15-phase20-erp-receipt-intake-design.md`](./2026-05-15-phase20-erp-receipt-intake-design.md)
- **데이터 모델**: [`./2026-05-15-phase20-erp-data-model.mmd`](./2026-05-15-phase20-erp-data-model.mmd)

## 한눈에 보기

영수증 사진 한 장을 올리면 AI가 거래처 / 일자 / 품목 / 금액을 자동으로 읽어내고, 사용자가 검토만 하면 **주문(Order)과 재고(InventoryMovement)가 한번에 등록**됩니다. 손으로 엑셀에 옮겨 적던 작업을 10초 안에 마칠 수 있습니다.

**언제 쓰나요?**
- 거래처에서 종이 영수증을 받았을 때
- 매장에서 카메라로 즉석 촬영해 등록하고 싶을 때
- 카드매출 전표를 정리할 때 (구매/판매 모두 가능)

**언제 못 쓰나요? (Phase 20 한계)**
- 이메일·PDF·여러 장 일괄 업로드 (Phase 21+)
- 한 영수증을 두 개의 주문으로 쪼개기 (지원 안 함 — 영수증 1장 = 주문 1건)

## 2. 검토 페이지에서 무엇을 하나요? (`/erp/intake/[draftId]`)

화면은 좌우 2단입니다.

| 좌측 | 우측 |
|------|------|
| 업로드한 영수증 이미지 미리보기 (확대/축소) | AI 추출 결과 편집 폼 |

### 필드별 편집 포인트

| 필드 | AI가 채움 | 사용자가 확인할 점 |
|------|----------|-------------------|
| **유형** | purchase / sale / unknown | unknown이면 반드시 구매/판매 중 선택 |
| **일자** | YYYY-MM-DD | 영수증 발급일과 일치하는지 |
| **거래처명** | 영수증의 가맹점명 | 표시용 snapshot으로 저장됨 |
| **거래처 (선택)** | 매칭 후보 top-3 | 기존 거래처와 일치하면 선택, 새로운 곳이면 비워두기 |
| **품목 (표)** | name / qty / unitPrice / unit | 단위(개/박스/kg)는 영수증에 없으면 직접 입력 |
| **품목 매칭** | 기존 Product 후보 top-3 + 점수 | 점수가 높으면 선택, 낮으면 "신규 등록" 체크 |
| **소계 / 세금 / 합계** | 영수증 숫자 그대로 | 금액 합이 안 맞으면 빨간 경고 표시 |

> TODO: 스크린샷 추가 (별도 PR) — 검토 페이지 좌우 분할

### 거래처 자동완성

거래처명에 타이핑하면 기존 `Client`(CRM)에서 후보가 떠오릅니다. **Phase 20에서는 거래처를 자동으로 연결하지 않습니다** — `counterpartyId`는 free-form snapshot이라 사용자가 직접 선택해야 저장됩니다. (CRM Client와 ERP 거래처는 의미가 다를 수 있어 의도적으로 분리.)

### 상품 매칭 + "신규 등록" 토글

품목 행마다 매칭 후보(top-3)와 점수가 표시됩니다.

- **점수 ≥ 0.6 (초록)**: 기존 Product와 같은 것이 거의 확실 → 후보 선택
- **점수 < 0.6 (노랑)**: 비슷하지만 다를 수 있음 → 직접 선택하거나 신규 등록
- **매칭 후보 없음**: "신규 등록" 토글을 체크하면 등록 시 Product가 자동 생성됨

> "신규 등록" 토글을 켜지 않으면 그 품목은 **ad-hoc 라인**으로 저장됩니다 — 주문(Order)에는 들어가지만 재고(InventoryMovement)는 발생하지 않습니다. 재고 관리가 필요한 품목은 반드시 토글을 켜거나 기존 Product를 선택하세요.

## 3. OCR 신뢰도가 낮다면?

AI는 0.0 ~ 1.0의 **신뢰도(confidence)** 점수를 함께 반환합니다.

- **≥ 0.8**: 보통 그대로 OK. 한 번 훑어보고 등록.
- **0.6 ~ 0.8**: 노란 배너로 "검토 권장" 표시. 일자/금액 위주로 재확인.
- **< 0.6**: 빨간 배너로 "수기 입력 권장". 영수증이 흐릿하거나 손글씨인 경우. 필요하면 모든 필드를 직접 입력해도 됩니다.

> 신뢰도가 낮아도 **등록을 막지는 않습니다**. 최종 책임은 사용자에게 있으므로, AI가 못 읽은 부분만 채워 넣고 진행하시면 됩니다.

## 4. 등록 후 어떻게 되나요?

**등록** 버튼을 누르면 한 번의 트랜잭션으로 다음이 모두 처리됩니다.

1. **IntakeDraft.status**: PENDING → CONFIRMED
2. **Order 생성** (구매면 PURCHASE, 판매면 SALE / status: CONFIRMED)
3. **OrderItem** N개 생성 (각 품목별 1라인)
4. **InventoryMovement** 자동 발생 (productId가 있는 라인만)
   - 구매(PURCHASE) → IN (재고 +)
   - 판매(SALE) → OUT (재고 −)
5. **신규 등록 토글된 품목**은 Product가 자동 생성됨 (sku가 있으면 upsert, 없으면 name+orgId로 dedup)
6. IntakeDraft에 confirmedOrderId 기록 (한 draft는 정확히 한 Order로만 연결됨)

등록 직후 자동으로 **주문 상세 페이지**(`/erp/orders/[orderId]`)로 이동합니다. 영수증 이미지는 출처(source) 링크로 다시 열 수 있습니다.

> **멱등성(idempotency)**: 등록 버튼을 더블클릭해도 두 번째 요청은 **409 Conflict**로 거부됩니다. 주문은 정확히 1건만 생성됩니다.

## 5. 취소는 어떻게 하나요?

잘못 등록한 영수증을 되돌려야 할 때:

1. **주문 상세 페이지** 열기 (`/erp/orders/[orderId]`)
2. 우상단 **취소** 버튼 클릭
3. 확인 모달에서 사유 입력 → 확정

자동 처리되는 일:
- Order.status → CANCELLED
- **역방향 InventoryMovement 자동 생성** (IN이었으면 OUT, 그 반대)
- 결과적으로 재고는 등록 이전 상태로 복원됨
- IntakeDraft는 그대로 (감사 추적용)

> 영수증 자체를 폐기하려면 **검토 페이지에서 "폐기"** 버튼을 누르세요 — IntakeDraft.status = DISCARDED, Blob 이미지는 다음 cron(일 1회)에서 정리됩니다.

## 7. Multi-org 환경에서의 동작

컨설팅 펌이 여러 managed org를 운영하는 경우(예: FlowCoder Inc. ↔ 고객사 5곳):

- 상단 **조직 스위처(topbar)**에서 active tenant 변경
- ERP 모든 데이터는 active tenant 기준으로만 보이고, 등록도 active tenant 아래로 들어갑니다
- 영수증 이미지의 Blob 경로에도 `<orgId>`가 포함되어 테넌트 간 누출이 발생하지 않습니다

> ⚠️ **주의**: 영수증을 업로드하기 전에 active tenant가 의도한 조직인지 반드시 확인하세요. 잘못된 조직에 등록되면 취소 → 올바른 조직에서 재등록이 필요합니다.

## FAQ

**Q. 여러 영수증을 한 번에 등록할 수 있나요?**
A. Phase 20에서는 1장씩만 가능합니다. 일괄 업로드(bulk import) 채널은 Phase 21+ 로드맵입니다.

**Q. 이메일이나 PDF로 받은 영수증은요?**
A. Phase 20은 이미지(JPG/PNG/WebP/HEIC)만 지원합니다. 이메일/PDF 채널은 Phase 21+ 입니다. 임시 우회법: PDF를 캡처해서 이미지로 변환 후 업로드.

**Q. AI가 잘못 인식한 정보는 어떻게 학습되나요?**
A. 현재는 자동 학습되지 않습니다 (Claude Vision은 stateless). 사용자가 수정한 데이터를 향후 fine-tuning이나 prompt 보강에 활용할 계획입니다 (Phase 21+).

**Q. 영수증 이미지는 얼마나 보관되나요?**
A. **5년** 입니다. 한국 세무 보관 의무(소득세법 §160, 부가가치세법 §32)에 따른 것입니다. 폐기한 영수증과 30일 이상 검토 안 된 PENDING 영수증은 일 1회 cron에서 자동 삭제됩니다. 자세한 정책은 [`./2026-05-15-phase20-blob-lifecycle.md`](./2026-05-15-phase20-blob-lifecycle.md) 참조.

**Q. 영수증 이미지 URL을 누가 볼 수 있나요?**
A. Phase 20 MVP는 Vercel Blob의 public scope를 사용하지만, URL에 12자 이상 랜덤 토큰이 포함되어 실질적으로 추측 불가합니다. 외부 검색엔진 색인은 차단됩니다. signed URL(짧은 만료)로의 전환은 Phase 21+ 입니다.

**Q. 거래처(counterparty)는 CRM의 Client와 어떻게 다른가요?**
A. Phase 20에서는 의도적으로 분리되어 있습니다 — Order.counterpartyId는 FK 없는 free-form 문자열로 시작합니다. CRM Client는 컨설팅 영업 관점의 고객이고, ERP 거래처는 거래 관점의 매입/판매 상대라 의미가 다를 수 있어서입니다. Phase 21에서 `ErpCounterparty` 모델로 분리될 예정입니다.

**Q. 신뢰도가 낮아도 등록되나요?**
A. 네. AI 신뢰도는 차단 기준이 아니라 **사용자 검토 강도**를 안내하는 정보입니다. 최종 책임은 사용자에게 있습니다.

**Q. 같은 영수증을 실수로 두 번 등록하면?**
A. 같은 draftId를 두 번 등록하려 하면 두 번째 요청은 409 Conflict로 거부됩니다(IntakeDraft.confirmedOrderId @unique). 하지만 사진을 두 번 업로드한 경우(별개의 draftId)는 시스템이 중복을 자동 감지하지 못합니다 — 사용자가 영수증 목록에서 확인 후 폐기해주세요.

**Q. 멀티 창고(multi-warehouse) 지원하나요?**
A. Phase 20은 단일 창고 모델입니다. 멀티 창고와 FIFO/이동평균 평가법은 Phase 21+ 입니다.

**Q. 재고가 마이너스로 떨어지면 어떻게 되나요?**
A. 별도 차단 없이 그대로 기록됩니다(논리적으로는 "외상 출고" 등이 가능). 음수 재고 alert/차단은 Phase 21+ 후속 작업입니다.

## 트러블슈팅

| 증상 | 원인 / 해결 |
|------|-------------|
| 업로드가 즉시 4xx 에러 | 권한 부족(`erp:write` 없음) 또는 파일 크기 초과. 설정 → 권한 확인. |
| OCR 처리 후 검토 페이지로 이동 안 됨 (PENDING + errorMsg) | Claude Vision provider 일시 오류. 검토 페이지의 **재시도** 버튼 클릭. |
| 모든 필드가 비어있고 raw text만 보임 | JSON 파싱 2회 실패. 수기 입력 fallback — 모든 필드 직접 채워서 등록. |
| 등록 버튼 눌렀는데 "이미 처리된 영수증입니다" (409) | 더블클릭 또는 다른 탭에서 이미 등록됨. 주문 목록에서 확인. |
| 등록 후 재고가 안 늘어남 | 해당 품목이 ad-hoc 라인(productId=null)으로 들어감 — "신규 등록" 토글이 꺼져 있었거나 기존 Product를 선택하지 않은 경우. 취소 후 재등록. |
